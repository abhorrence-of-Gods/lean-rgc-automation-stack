## 5. The sole functional repair: cache constants and stabilize one operation

The performance repair is deliberately narrower than a general guard rewrite.
For one active strict GuardPolicy, the guard may precompute exactly:

1. the ordered 17 pairs
   canonical-denied-path and prefix-flag derived from DENYLIST_ROWS;
2. the canonical EMIT_ROOT; and
3. the canonical seven CLOSEOUT_ARTIFACTS as an immutable ordered tuple.

These values are mode-independent authority constants.  They may be compiled
once before the active installation's wrappers and audit hook begin handling
semantic file operations, retained only for that exact canonical-root
installation, and discarded when its owning GuardHandle closes.  A strict
same-root policy may consult the same constant table when `_check_path`
re-evaluates SEMANTIC, EMIT, or CLOSEOUT permissions, but `install_guard` and
`require_active_guard` still require the exact active policy and reject a
nested installation with another mode.  The representation must be bounded
and immutable.  It may not be loaded from a file, environment
variable, attestation, caller argument, previous process, previous
installation, pickle, pytest cache, pycache, or network.

Every caller-supplied path remains freshly passed through the existing
_canonical_path on every operation.  In particular, R4 may not cache:

- a caller path, os.PathLike object, file descriptor, or lexical string;
- an allowed/denied decision;
- realpath, current-working-directory, existence, stat, symlink, junction,
  case, mode, read/write, or enumeration results;
- import or capability decisions; or
- entries keyed by arbitrary caller input.

Within one wrapper operation, however, the caller operand is materialized
exactly once to a strict immutable `str`, `bytes`, or integer descriptor.
That same stable operand is supplied to `_check_path` and to the underlying
filesystem primitive.  A `Path` method may rebuild one exact platform `Path`
from that single materialization before both uses.  A two-path operation does
this independently once for source and once for target.  This closes the
audited check/use bypass in which a hostile `PathLike.__fspath__` returned a
safe path for authorization and a denied path for the syscall.  The stable
operand lives only for that call: it is not a caller-path, realpath, or
decision cache and is never reused across operations.

There is no unbounded lru/dict cache and no cross-root or cross-installation
reuse.  An absent, malformed, mutable, wrong-root, reordered,
wrong-cardinality, duplicate, or forged compiled scope fails closed before it
can authorize an operation.  The SEMANTIC, EMIT, and CLOSEOUT exceptions are
unchanged: SEMANTIC denies every matching row; EMIT permits only write-only,
non-enumerating creation of the exact root/seven artifacts; CLOSEOUT permits
only read-only, non-enumerating access to the exact seven artifacts.

No wrapper, audit event, forbidden import prefix, denial row, CI setup rule,
static source scan, control attestation, or active-guard identity check may be
removed.  Requested-path canonicalization remains the alias/retarget defense;
the cache removes only the redundant canonicalization of frozen constants.

## 6. Frozen guard attacks

The existing five top-level identity-test names remain exact.  The following
cache attacks are incorporated into those existing tests without adding a
sixth top-level identity test:

1. **Independent constant reconstruction.**  The compiled ordered deny rows,
   all 17 prefix bits, EMIT root, and seven artifact members must equal a
   fresh literal-independent reconstruction; omission, duplication,
   reordering, changed prefix, wrong root, wrong mode, or an eighth artifact
   fails.
2. **No repeated authority work.**  Instrumented consecutive checks prove
   that frozen deny rows/artifacts are not canonicalized again after the
   active scope is compiled, while the supplied operand is canonicalized once
   afresh for each check.
3. **Mutable PathLike attacks.**  Across two operations, the same strict
   os.PathLike instance returns a safe path first and a denied path second;
   the second operation is denied and `__fspath__` is called again.  Within
   one wrapper operation, a flipping instance that would return safe on its
   first conversion and denied on its second is converted exactly once, so
   authorization and the underlying syscall receive the identical operand.
   Side effects that try to replace the active table or mutate the strict
   policy during conversion fail closed and cannot alter the current check.
4. **Retarget-after-warm attack.**  The same lexical operand resolves safely
   once and to a denied row after the resolver is changed.  The second
   operation must be denied; no cached caller realpath or decision may win.
5. **Relative-path/CWD attack.**  The same relative spelling is checked under
   two independently controlled absolute bases, the second landing in a
   denied row.  The second operation must be denied.
6. **Mode/cache-separation matrix.**  The same exact artifact path is denied
   under SEMANTIC; under EMIT only write-only non-enumeration is allowed; under
   CLOSEOUT only read-only non-enumeration is allowed.  Read+write,
   enumeration, root read, sibling, prefix-confusable, and unlisted eighth
   artifact attacks remain denied.  A compiled scope from another root or
   installation cannot be substituted.  Same-root mode checks reuse only the
   frozen constant table; mode and operation permissions are evaluated anew
   on every call.
7. **Alias matrix.**  Absolute, relative, slash/backslash, case-variant, and
   at-or-below spellings of every applicable frozen row remain denied after
   cache warm-up.  Exact-file rows do not accidentally become prefix rows,
   and prefix rows do not admit descendants.
8. **Bounded-cache attack.**  Checking many distinct safe caller paths does
   not grow the compiled authority scope or create caller-keyed entries.
   Integer file descriptors preserve their existing non-path handling and do
   not enter a cache.
9. **Lifecycle attack.**  Closing the owning handle clears the active scope.
   Reinstallation constructs a fresh scope; stale handles, a second policy,
   or a second active installation cannot reuse or replace it.  If the active
   table, token, digest, or policy fingerprint is found missing or corrupt at
   owner close, the guard restores its installed wrappers and clears every
   active field before reporting the fail-closed lifecycle error; it may not
   strand a half-active installation behind an already-closed handle.

The static scanner must bind the exact implementation bytes and retain all
pre-existing process, socket, dynamic-import, native Lean, official
transport, quarantine, protected path, and mutation attacks.
